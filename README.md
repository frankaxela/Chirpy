# Chirpy

Chirpy is a Twitter-like social network HTTP API written in Go, built as a guided project for [Boot.dev](https://www.boot.dev). Users can register, log in, and post short messages ("chirps"), with JWT-based authentication, refresh tokens, and a payment-provider webhook that upgrades users to Chirpy Red membership.

![Chirpy logo](assets/logo.png)

## Features

- User registration and login with argon2id password hashing
- JWT access tokens (1 hour expiry) plus revocable refresh tokens (60 days)
- Create, list, fetch, and delete chirps (max 140 characters, profanity filtered)
- Chirpy Red membership upgrades via the Polka payment webhook, authenticated by API key
- Static file server with hit metrics and a dev-only reset endpoint

## Requirements

- Go 1.26+
- PostgreSQL
- [goose](https://github.com/pressly/goose) (database migrations)
- [sqlc](https://sqlc.dev) (only if you change the SQL queries or schema)

## Setup

1. Create a PostgreSQL database:

   ```sql
   CREATE DATABASE chirpy;
   ```

2. Create a `.env` file in the repo root:

   ```env
   DB_URL="postgres://user:password@localhost:5432/chirpy?sslmode=disable"
   PLATFORM="dev"
   SECRET="<random string used to sign JWTs>"
   POLKA_KEY="<API key expected from the Polka webhook>"
   ```

   `PLATFORM="dev"` enables the destructive `POST /admin/reset` endpoint; leave it unset in production.

3. Run the migrations:

   ```bash
   goose -dir sql/schema postgres "$DB_URL" up
   ```

4. Build and run the server (listens on `:8080`):

   ```bash
   go build -o chirpy . && ./chirpy
   ```

## API

### Health and admin

| Method | Path | Description |
| --- | --- | --- |
| GET | `/api/healthz` | Health check |
| GET | `/app/` | Static file server |
| GET | `/admin/metrics` | File server hit count |
| POST | `/admin/reset` | Delete all users and chirps (dev only) |

### Users and auth

| Method | Path | Auth | Description |
| --- | --- | --- | --- |
| POST | `/api/users` | — | Register with `{"email", "password"}` |
| PUT | `/api/users` | Bearer JWT | Update email and password |
| POST | `/api/login` | — | Log in; returns user, access token, and refresh token |
| POST | `/api/refresh` | Bearer refresh token | Exchange refresh token for a new access token |
| POST | `/api/revoke` | Bearer refresh token | Revoke a refresh token |

### Chirps

| Method | Path | Auth | Description |
| --- | --- | --- | --- |
| POST | `/api/chirps` | Bearer JWT | Create a chirp with `{"body"}` (max 140 chars) |
| GET | `/api/chirps` | — | List all chirps |
| GET | `/api/chirps/{chirpID}` | — | Get a single chirp |
| DELETE | `/api/chirps/{chirpID}` | Bearer JWT | Delete your own chirp |

### Webhooks

| Method | Path | Auth | Description |
| --- | --- | --- | --- |
| POST | `/api/polka/webhooks` | `ApiKey <POLKA_KEY>` | Handles `user.upgraded` events; upgrades the user to Chirpy Red |

## Development

Run the tests:

```bash
go test ./...
```

Database access code in `internal/database/` is generated by sqlc from `sql/queries/` and `sql/schema/`. After editing the SQL, regenerate it with:

```bash
sqlc generate
```
