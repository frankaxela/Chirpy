# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Chirpy is a Twitter-like HTTP API server in Go (Boot.dev guided project). Single binary, PostgreSQL backend, JWT auth with refresh tokens, and a Polka payment webhook for "Chirpy Red" membership upgrades.

## Commands

```bash
go build -o chirpy .        # build (binary name is gitignored)
go run .                    # run the server on :8080
go test ./...               # run all tests
go test ./internal/auth/ -run TestName   # run a single test
sqlc generate               # regenerate internal/database/ from sql/ (run after editing queries/schema)
goose -dir sql/schema postgres "$DB_URL" up   # apply migrations
```

The server reads config from `.env` (gitignored, loaded via godotenv): `DB_URL`, `PLATFORM`, `SECRET` (JWT signing key), `POLKA_KEY`. `PLATFORM=dev` enables the destructive `POST /admin/reset` endpoint.

## Architecture

- `main.go` — everything HTTP: route registration (Go 1.22+ `net/http` method+path patterns on `http.ServeMux`), all handlers as methods on `apiConfig` (holds `*database.Queries`, secrets, hit counter), and the JSON response helpers `respondWithJSON`/`respondWithError`. Response shapes are the `chirpResponse`/`userResponse` structs at the bottom.
- `internal/auth/` — hand-written auth: argon2id password hashing, HS256 JWT create/validate (1h expiry, subject = user UUID), refresh token generation (random hex, 60-day expiry stored in DB), and `Bearer`/`ApiKey` Authorization-header parsing.
- `internal/database/` — **generated by sqlc, do not edit by hand.** Source of truth is `sql/queries/*.sql` (sqlc-annotated queries) and `sql/schema/*.sql` (numbered goose migrations). To change DB access: edit the SQL, add a new numbered migration for schema changes, then run `sqlc generate`.

Auth flow: `POST /api/login` returns an access JWT plus a refresh token; `POST /api/refresh` exchanges a valid, unrevoked refresh token for a new JWT; `POST /api/revoke` revokes it. Protected endpoints validate the Bearer JWT and use its subject as the acting user ID. The Polka webhook authenticates with `Authorization: ApiKey <POLKA_KEY>` instead of a JWT.

## Workflow

- Create a feature branch per assignment and open a PR against `main`; never commit directly to `main`.
- Tests must cover every branch, error path, and edge case of the code under test (see `internal/auth/password_test.go` for the expected style).
